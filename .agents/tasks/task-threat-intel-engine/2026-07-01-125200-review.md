# Threat Intelligence Engine (v2)

A complete threat intel module comprising IOC lifecycle management, multi-format feed parsers (STIX 2.1, CSV, plain text, OpenIOC XML), a matching engine with IP trie and domain suffix lookup, MITRE ATT&CK mapping, CVE database, campaign correlation, threat scoring, report generation, and enrichment service. The v1 review's top 5 issues have been addressed: `matchAll` uses a public-locking/private-impl split, feed ingestion is incremental, `initialized_` is `std::atomic<bool>`, STIX pattern extraction locates the value after `= '`, and domains are pre-lowercased at insert time.

**Watch for:** The incremental ingestion uses `ioc.id > sizeBefore` to detect new IOCs, which conflates the IOC count with the ID sequence and will mis-identify old IOCs as new after any deletion (likely). `processIndicator` calls `getActiveIOCs()` then iterates all IOCs in `findRelatedIOCs` on every single match, making per-indicator processing O(n) in total IOC count (confirmed). `ThreatScoringEngine` and `EnrichmentService` have no thread synchronization, yet are called from the thread-safe facade (possible race if callers mutate configuration concurrently with processing).

## High-level view

The v1 fixes are correctly applied. The new concerns are narrower: the incremental ingestion heuristic for detecting "new" IOCs uses `iocManager_->size()` as an ID threshold, but `size()` tracks map occupancy while IDs are monotonically increasing — any prior deletion breaks the assumption. The `processIndicator` hot path copies the entire IOC vector and linearly scans it for correlation on every call, making the matching engine's O(1) trie/hashmap lookups irrelevant to overall latency. `ThreatScoringEngine` exposes mutable `configureWeights`/`setSourceReliability` methods with no internal locking, creating a data race surface if the facade is used from multiple threads.

## Incremental ingestion uses IOC count as ID threshold

In `ThreatIntelEngine::ingestFeed`:

```cpp
size_t sizeBefore = iocManager_->size();
// ... processFeedData adds IOCs ...
auto allIOCs = iocManager_->getActiveIOCs();
for (const auto& ioc : allIOCs) {
    if (ioc.id > sizeBefore) {
        iocMatcher_->addIOC(ioc);
    }
}
```

`IOCManager::size()` returns `iocs_.size()` (the current map count), while IDs are assigned from `nextId_` which only ever increases. After calling `removeIOC` once, `size()` is permanently less than `nextId_ - 1`. The filter `ioc.id > sizeBefore` will then match IOCs that existed before this ingestion, causing duplicate entries in the matcher's data structures.

Fix: have `processFeedData` return the IDs it added (it already knows them since `addIOC` returns each ID), or expose a `nextId()` method on IOCManager and snapshot it before ingestion starts.

## processIndicator creates full IOC copies on every call

```cpp
ProcessResult ThreatIntelEngine::processIndicator(const std::string& value) {
    auto matches = iocMatcher_->matchAll(value);
    // ...
    auto allIOCs = iocManager_->getActiveIOCs();  // copies entire vector
    auto relationships = correlationEngine_->findRelatedIOCs(
        bestMatch->matchedIOC.id, allIOCs);
```

`getActiveIOCs()` acquires a shared lock and copies every non-expired IOC into a new vector. `findRelatedIOCs` then iterates this vector comparing source strings, computing time differences, and comparing tag vectors. The matching itself (via trie and hash maps) is O(1)/O(m), so the correlation dominates total latency at scale.
