# Threat Intelligence Engine for ThreatOne

A complete threat intelligence module with IOC lifecycle management, multi-format feed ingestion (STIX 2.1, CSV, plain text, OpenIOC XML), matching engine with IP trie and domain suffix lookup, MITRE ATT&CK mapping, CVE database, campaign correlation, threat scoring, report generation, and enrichment service. The module integrates into SecurityEngine via a `checkThreatIntel` delegation method and ships with 10 test files covering all components. The design is a facade (`ThreatIntelEngine`) that owns and wires together 9 sub-components, each independently thread-safe with `shared_mutex`.

**Watch for:** IOCMatcher `matchAll` acquires the same `shared_mutex` four times recursively, which will deadlock on non-recursive shared mutexes (confirmed). Domain matching iterates the full domain map on every query, degrading to O(n) with large indicator sets (confirmed). The STIX pattern extractor uses `rfind('\'')` which will mis-extract values from patterns containing quoted property paths (likely). `ThreatIntelEngine::initialized_` is a plain `bool` read/written from potentially different threads without synchronization (likely).

## High-level view

The module uses a layered ownership model: `ThreatIntelEngine` facade allocates all nine sub-components on `initialize()` and provides `processIndicator`, `ingestFeed`, and `generateReport` as the primary entry points. SecurityEngine constructs the facade unconditionally in its own constructor, so creating any SecurityEngine instance now initializes the entire MITRE matrix (25 techniques, 11 tactics) and correlation infrastructure at startup.

IOC matching relies on three data structures: an `unordered_map` for exact (hash/email) lookups, a character-level trie for IP prefix matching, and a flat map with linear scan for domain suffix matching. The trie is appropriate for IP prefixes, but domain matching has no structural optimization and scans every stored domain on each query.

Feed ingestion flows through format-specific parsers to `IOCManager::addIOC`, followed by a full matcher rebuild (`clear()` + `loadIOCs()`). This rebuild strategy means ingesting feeds is O(n) in total IOC count regardless of how many new indicators arrived, and creates a window where the matcher is empty.

The threat scoring formula multiplies base confidence by source reliability and age decay, then adds a correlation boost capped at 20 points. The 45-day half-life and 0.1 floor on decay are hardcoded constants not exposed through `configureWeights`, despite that method existing for other parameters.

## Recursive lock acquisition in IOCMatcher

`IOCMatcher::matchAll` acquires no lock itself but calls four public methods (`matchExact`, `matchDomain`, `matchIP`, `matchURL`), each of which acquires `std::shared_lock lock(mutex_)`. The C++ standard says acquiring a `shared_lock` on a `shared_mutex` that the calling thread already holds as shared is undefined behavior.

In practice, many Linux implementations (libstdc++ with pthreads) happen to allow recursive shared locks, so this may work today but is a portability and correctness landmine. The fix is straightforward: extract lockless `matchExact_impl`, `matchDomain_impl`, etc., have the public methods acquire the lock and delegate, and have `matchAll` acquire once and call the `_impl` methods directly.

The same issue exists in `matchURL`, which duplicates the domain-matching logic inline rather than calling `matchDomain` (specifically to avoid this problem for domains) but still acquires the lock separately from the caller path through `matchAll`.

## Domain matching scalability

```cpp
for (const auto& [storedDomain, ids] : domainMap_) {
    std::string lowerStored = storedDomain;
    std::transform(lowerStored.begin(), lowerStored.end(), lowerStored.begin(),
                   [](unsigned char c) { return static_cast<char>(std::tolower(c)); });
    // ... suffix comparison
}
```

Every `matchDomain` call lowercases every stored domain and compares. With 10K domain IOCs, this is 10K string allocations and comparisons per query. A reversed-domain trie (store "com.evil" as a trie path) enables O(m) lookups where m is the query domain length. At minimum, storing pre-lowercased domains at insert time eliminates the per-query transform.

## Feed ingestion triggers full rebuild

In `ThreatIntelEngine::ingestFeed`:

```cpp
if (result.isOk()) {
    iocMatcher_->clear();
    iocMatcher_->loadIOCs(*iocManager_);
}
```

Since `IOCMatcher::addIOC` already exists and correctly routes to the appropriate data structure, the facade could call it for each newly-ingested IOC instead of rebuilding. The current approach also creates a window where the matcher is empty between `clear()` and `loadIOCs()` â€” any concurrent `processIndicator` call during that window would find no matches.

## STIX pattern value extraction

```cpp
auto firstQuote = pattern.find('\'');
auto lastQuote = pattern.rfind('\'');
outValue = pattern.substr(firstQuote + 1, lastQuote - firstQuote - 1);
```

For a pattern like `[file:hashes.'SHA-256' = 'abc123']`, `firstQuote` points to the quote before `SHA-256`, not the quote before the value. The extracted value would be `SHA-256' = 'abc123` rather than `abc123`. The test data works around this by using `SHA256` without the dash/quotes in the property path. Production STIX feeds commonly use the quoted property form `hashes.'SHA-256'`.

## Correlation engine's source-only grouping

`correlateIOCs` groups IOCs exclusively by `source` field. The default `CorrelationRule` configured by the facade has an empty `iocTypes` list (matches all types), so any two IOCs from the same source within an hour form a "campaign". This produces high false-positive campaign counts in environments with busy feed sources. A useful mitigation: require rules to specify at least two distinct IOC types, or weight the correlation score by type diversity.

## Test coverage gaps

No `test_threat_intel_engine.cpp` exists. The integration path `processIndicator` â†’ match â†’ score â†’ correlate is untested at the unit level. The `checkThreatIntel` SecurityEngine method is also not unit-tested. Thread safety is implied by the `shared_mutex` usage throughout but no test actually exercises concurrent access (multiple threads reading/writing simultaneously).
