# Enterprise identity, licensing, fleet, and API modules

This change adds ~8,500 lines across four module expansions: identity (RBAC, authentication, SSO stubs, LDAP connector, cloud identity providers, audit logger), licensing (key validator, feature gates), fleet (endpoint management, policy engine, remote commands, staged deployments), and API (HTTP router, REST controllers, API key/JWT auth, webhooks). Each module follows the existing project pattern of interface + concrete implementation with in-memory state. The code compiles and passes 653 tests.

**Watch for:** MFA verification accepts any 6-character string as valid (confirmed). Webhook secret is transmitted in a header rather than used for HMAC signing (confirmed). Audit chain integrity breaks after retention enforcement (confirmed). Identity module classes lack mutex protection, creating data race potential when used from API handlers (likely).

## High-level view

The authentication subsystem implements a hand-rolled SHA-256 with 1000-round key stretching. The MFA implementation is a stub that grants verification to any 6-digit code without validating the TOTP secret, meaning MFA is effectively a no-op security control.

The licensing module uses a djb2-based checksum for license key validation, which is trivially reversible. Feature gates propagate tier changes immediately to all registered features with degradation callbacks for the UI layer.

Fleet management is the most complete new subsystem. Heartbeat timeout detection, staged rollouts with auto-rollback thresholds, and policy versioning with compliance checking are all implemented with proper mutex protection. The deployment manager's `advanceStage` updates endpoint versions positionally rather than tracking which specific endpoints succeeded.

The API module delivers a functional path-parameter router, middleware chain, and JWT generation/validation. Controllers return hardcoded stub data rather than integrating with actual subsystems. The webhook manager sends the subscription secret as a plain header value instead of computing an HMAC signature.

<summary>Issues (8)</summary>

1. **MFA accepts any 6-character code** — `AuthenticationManager::verifyMFA` marks MFA as verified for any code with `length() == 6`, ignoring the stored `mfaSecret`. Implement actual TOTP validation or at minimum compare against the secret.
2. **Webhook secret sent in cleartext header** — `WebhookManager::deliverWebhook` sets `X-Webhook-Secret` to the raw secret. Use the secret as an HMAC key to sign the payload, emitting only the signature.
3. **Audit chain breaks on retention enforcement** — `AuditLogger::enforceRetention` erases entries from the vector, breaking `previousHash` linkage for subsequent entries. Chain verification will fail after any retention pass.
4. **Identity module has no thread safety** — `RBACManager`, `AuthenticationManager`, and `AuditLogger` use no synchronization. If accessed from API request handlers (which do use mutexes), data races will occur.
5. **Rate-limit state is process-local** — `ApiAuth::rateLimits_` is an in-memory map. If ThreatOne restarts, rate limits reset. Document this limitation or add a persistence interface.
6. **DeploymentManager stage success tracking is positional** — `advanceStage` updates the first N endpoints regardless of which actually succeeded. Track success per endpoint ID.
7. **Controllers return hardcoded data** — All REST controllers return static JSON rather than integrating with their respective modules. These need wiring to real data sources before they're useful.
8. **License checksum is trivially reversible** — The djb2-based 5-character checksum can be brute-forced in milliseconds. Acceptable for a first implementation but not a barrier to key forgery.

## MFA verification is a no-op

`AuthenticationManager::verifyMFA` (AuthenticationManager.cpp) accepts any string of exactly 6 characters as a valid MFA code:

```cpp
bool AuthenticationManager::verifyMFA(const std::string& token, const std::string& code) {
    auto it = sessions_.find(token);
    if (it == sessions_.end()) return false;
    if (code.length() == 6) {
        it->second.mfaVerified = true;
        return true;
    }
    return false;
}
```

The stored `mfaSecret` is never consulted. The method returns `true` and sets `mfaVerified = true`, so downstream code checking MFA status will believe verification succeeded. This is fail-open: enabling MFA provides zero additional security until TOTP validation is implemented.

## Webhook secret exposure

In `WebhookManager::deliverWebhook`, the subscription secret is placed directly in the `X-Webhook-Secret` header:

```cpp
headers["X-Webhook-Secret"] = sub.secret;
```

Standard webhook implementations use the secret as an HMAC-SHA256 key to sign the payload body, then send only the signature. Sending the raw secret means any intermediary can extract it. The `WebhookSender` callback interface makes it easy to fix: compute `HMAC(secret, payload)` and send that instead.

## Audit chain integrity vs. retention

`AuditLogger::enforceRetention` iterates `entries_` and erases entries that exceed `maxAge`. After removal, the `previousHash` field of the surviving entry still references the hash of a deleted entry. `verifyChainIntegrity()` will fail because `entries_[i].previousHash != entries_[i-1].hash` when the predecessor was removed.

Two possible fixes: mark entries as "archived" without removing them from the vector (preserves tamper-evidence), or re-link the chain after eviction (sacrifices tamper-evidence but keeps the API consistent).

## Identity module thread safety gap

Fleet and API classes consistently acquire `std::mutex` at method boundaries. The identity module classes (`RBACManager`, `AuthenticationManager`, `AuditLogger`) do not use any synchronization. If identity components are accessed from multiple threads — likely when API request handlers call `validateSession` or `checkPermission` concurrently — data races on `sessions_`, `credentials_`, and `entries_` will occur. Either add mutexes to identity classes or document that they must be externally synchronized.

## Deployment stage tracking fidelity

`DeploymentManager::advanceStage` receives `successes` and `failures` counts but has no way to know *which* endpoints succeeded:

```cpp
for (const auto& epId : stage.endpointIds) {
    if (updated < successes) {
        endpointVersions_[epId] = {epId, it->second.version, ...};
        updated++;
    }
}
```

This assigns the new version to the first `successes` endpoints in the list regardless of actual outcome. The interface should accept a set of succeeded endpoint IDs.
