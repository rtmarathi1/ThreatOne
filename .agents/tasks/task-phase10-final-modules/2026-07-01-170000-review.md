# Nine stub modules replaced with in-memory implementations

Phase 10 replaces all remaining stub modules (XDR, SIEM, SOC, Cloud, Plugin, Updates, Sandbox, HIDS, Telemetry) with functional implementations backed by in-memory data structures. Each module follows the same architecture: a mutex-protected map-based store, sequential ID generation, and an expanded interface that extends the original stubs. The change adds ~9,300 lines across 9 modules with 229 tests covering the new functionality.

**Watch for:** SIEM correlation rules ignore their configured time window and match against the entire log store (confirmed). The `reloadPlugin` method has a TOCTOU gap between its existence check and its calls to `unloadPlugin`/`loadPlugin` (confirmed). Timestamp handling across XDR and SIEM uses thread-unsafe `std::mktime` and `std::gmtime` (confirmed). Atomic counters used alongside mutex-protected methods are unnecessary complexity that could cause ID gaps under contention (likely).

## High-level view

The nine modules share an identical structural pattern: a pure-virtual interface gains new methods, the concrete class holds `std::map` or `std::unordered_map` stores protected by a single `std::mutex`, IDs are generated by incrementing an `int`, and every public method acquires the lock then operates on the store. This uniformity makes the codebase navigable but means any architectural flaw is replicated nine times.

The SIEM engine's correlation rule evaluation scans all logs that match a condition string without actually constraining by the rule's `timeWindowSeconds`. The time window logic is stubbed out with an empty comment block. This means any threshold-based rule fires on historical totals rather than recent activity, producing false positives in any real usage.

The XDR engine implements proper multi-endpoint event correlation with severity escalation detection and cross-source alert fusion. The confidence scoring model is straightforward (additive per event count and source count), and the automated investigation pipeline generates findings from stored correlations. The implementation is functional but the timestamp comparison relies on `std::mktime` which is locale-dependent and not thread-safe.

The Plugin Manager's dependency resolver uses Kahn's algorithm for topological sort, correctly detecting circular dependencies. However, the missing-dependency check runs after the topological sort completes, meaning it can produce incorrect results when the graph itself is incomplete.

Test coverage spans all nine modules with 229 test cases total, but tests are entirely in-process using the concrete implementations directly. No concurrency tests exist despite the mutex usage, and no tests verify behavior under error injection or partial-failure scenarios.

<details>
<summary>Issues (7)</summary>

1. **SIEM time window ignored** — `evaluateRules()` matches against all stored logs regardless of `timeWindowSeconds`. Implement actual timestamp comparison to restrict matches to the configured window.
2. **Thread-unsafe time functions** — `std::mktime` (XDR) and `std::gmtime` (Updates) are not thread-safe. Replace with `gmtime_r`/`timegm` or use `<chrono>` throughout.
3. **Plugin reload TOCTOU** — `reloadPlugin` checks existence, releases the lock, then calls `unloadPlugin` and `loadPlugin` separately. A concurrent call could race between those steps. Consider holding the lock across the full operation or using an internal `_unlocked` variant.
4. **Atomic counters redundant** — CloudManager, XDREngine, and SOCManager use `std::atomic<int>` for ID counters but only ever increment them under the same mutex they use for everything else. This is harmless but misleading; plain `int` would be consistent with the modules that already use plain ints (HIDS, Sandbox, Plugin, Updates, Telemetry).
5. **Plugin dependency check ordering** — `resolveDependencies` runs the topological sort before checking whether required dependencies actually exist, which can return a valid order for an unresolvable graph.
6. **Cloud `downloadPolicies` builds JSON by hand** — String concatenation without escaping means policy names containing quotes or backslashes produce malformed JSON.
7. **No concurrency tests** — All nine modules use `std::mutex` but no test exercises multi-threaded access patterns.

## SIEM correlation ignores time window

The `evaluateRules()` method iterates over all stored logs and checks `matchesCondition()` against each. The `CorrelationRule::timeWindowSeconds` field is read nowhere in the evaluation loop. The comment block acknowledges this gap:

```cpp
// Simple window check: if we have timestamps, compare them
if (!logs_.empty() && !it->timestamp.empty()) {
    // We use the approach of checking all logs with matching timestamps
    // within the time window relative to the most recent matching log
}
```

This is dead code. The effect is that a rule with threshold=5 fires as soon as the cumulative total of matching logs across the engine's lifetime reaches 5, regardless of whether they occurred within the configured window. The tests pass because they ingest exactly the right number of logs and evaluate immediately, never testing the "below threshold within window but above threshold overall" scenario.

## Thread-unsafe timestamp handling

`XDREngine::eventsWithinTimeWindow` calls `std::mktime` which modifies the `tm` struct and uses global timezone state. In a multi-threaded context this is undefined behavior. The UpdateManager uses `std::gmtime` which returns a pointer to a static buffer; concurrent calls from different threads would corrupt each other's results. The HIDS and Sandbox modules sidestep this by using `std::chrono::system_clock::to_time_t` directly for simple epoch strings.

## Plugin reload race window

```cpp
bool PluginManager::reloadPlugin(const std::string& pluginId) {
    std::string path;
    {
        std::lock_guard<std::mutex> lock(mutex_);
        auto it = plugins_.find(pluginId);
        if (it == plugins_.end()) { return false; }
        path = pluginId;
    }
    // Lock released here
    if (!unloadPlugin(pluginId)) { ... }  // Re-acquires lock
    if (!loadPlugin(path)) { ... }        // Re-acquires lock
    ...
}
```

Between releasing the lock and calling `unloadPlugin`, another thread could unload or modify the same plugin. The `unloadPlugin` call would then either fail or operate on stale state. Since `loadPlugin` and `unloadPlugin` each independently acquire the mutex, the reload is not atomic.

## Redundant atomics mixed with mutex

CloudManager, XDREngine, and SOCManager declare ID counters as `std::atomic<int>` but only ever access them inside lock_guard scopes. The other six modules (HIDS, Sandbox, Plugin, Updates, Telemetry, SIEM) use plain `int` counters under the same pattern. This inconsistency suggests copy-paste divergence rather than a deliberate design choice. The atomics add no safety benefit when already under the mutex and could mislead a future maintainer into thinking lock-free access is safe.

## Cloud JSON construction

`CloudManager::downloadPolicies()` concatenates policy names directly into a JSON string without escaping:

```cpp
result += "{\"id\":\"" + id + "\",\"name\":\"" + policy.name + "\"}";
```

If `policy.name` contains a quote character, the output is malformed JSON. The project already links `nlohmann_json`; using it here would be both safer and shorter.

## Test coverage shape

229 test cases across 9 modules cover the happy-path CRUD operations well: create, retrieve, update, delete, list, search. Edge cases that are tested include duplicate prevention (HIDS path monitoring), state machine transitions (SOC case lifecycle rejects invalid transitions), and threshold boundaries (SIEM correlation below/above threshold). Not tested: concurrent access, error recovery after partial failures, behavior under memory pressure, and the actual time-window semantics of correlation rules.
