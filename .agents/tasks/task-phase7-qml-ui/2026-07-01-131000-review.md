# Complete QML UI layer for ThreatOne Enterprise (v2)

Phase 7 delivers the full QML presentation tier: a ThemeManager singleton with 100+ design tokens, 8 reusable widgets, 3 enhanced components (NavigationSidebar, TopBar, StatusBar), 10 pages, and 10 C++ ViewModels behind `#ifdef ENABLE_QT_BUILD` guards. The fix commit addressed the three most actionable issues from v1: widget/theme imports are now resolved via both relative `import` statements and `qmldir` module declarations, the sidebar's flat-index calculation uses pre-computed offsets instead of fragile parent traversal, and `App.qml` sample values were aligned with page stubs to eliminate the dual source of truth.

Watch for:
- **Threat Intel page path mismatch** (confirmed) — NavigationSidebar routes to `pages/threatintel/ThreatIntelPage.qml` but the file lives at `pages/threat_intel/ThreatIntelPage.qml`. This will produce a runtime load failure.
- **StackView duplicate-replace guard never fires** (confirmed) — `main.qml` compares `contentStack.currentItem.objectName` (e.g. `"dashboard"`) against `page` (e.g. `"pages/dashboard/DashboardPage.qml"`); the comparison always evaluates to false, so clicking the active sidebar item triggers a redundant page replace.

**Verdict**: NEEDS_CHANGES

## High-level view

A path typo remains in the sidebar model: the Threat Intel entry points to `pages/threatintel/` (no underscore) while the actual directory is `pages/threat_intel/`. This is a page-load failure at runtime for one of the ten navigation targets.

The page-replacement guard in `main.qml` uses `objectName` equality to skip redundant replaces, but `objectName` holds short identifiers like `"dashboard"` while the incoming `page` variable holds the file path `"pages/dashboard/DashboardPage.qml"`. The guard is dead code — every sidebar click triggers a full page destruction and reconstruction regardless of whether navigation actually changed.

ViewModel-to-QML wiring and test coverage remain absent, as acknowledged in v1. These are expected gaps given the Qt6-absent build environment.

## Threat Intel navigation path mismatch

The `NavigationSidebar` model at line 36 declares:

```qml
{ title: "Threat Intel", icon: "\u2604", page: "pages/threatintel/ThreatIntelPage.qml", badge: 0 }
```

The actual file system path is `qml/pages/threat_intel/ThreatIntelPage.qml` (with an underscore). Since `StackView.replace()` resolves paths relative to the QML file's location, and `main.qml` lives at `qml/main.qml`, the engine will attempt to load `qml/pages/threatintel/ThreatIntelPage.qml` — a nonexistent path. The other 9 navigation entries use the correct directory names.

## StackView duplicate-replace guard

In `main.qml`:

```qml
onPageSelected: function(page) {
    if (contentStack.currentItem && contentStack.currentItem.objectName === page)
        return
    contentStack.replace(page, {}, StackView.Transition)
    ...
}
```

The `objectName` property on each page is set to a short identifier (e.g. `"dashboard"`, `"scanner"`, `"threat_intel"`), while `page` is the file path (e.g. `"pages/dashboard/DashboardPage.qml"`). These will never be equal. Every sidebar click triggers a full page destruction and reconstruction even when the target page is already displayed.

A minimal fix: track the current page path in a property and compare against that, or extract a key from the path string.
