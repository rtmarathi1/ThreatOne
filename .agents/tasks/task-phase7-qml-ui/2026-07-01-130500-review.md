# Complete QML UI layer for ThreatOne Enterprise

Phase 7 delivers the full presentation tier: a design-token-based theming system (100+ tokens across dark/light themes), 8 reusable widgets, 10 production pages, and 10 C++ ViewModels compiled behind `#ifdef ENABLE_QT_BUILD` guards so the headless build remains green. The QML layer uses inline `QtObject` stubs as sample data sources, deferring real backend integration. Page routing runs through a `StackView` with transition animations and keyboard shortcuts (Ctrl+1..9 for page selection, Ctrl+B for sidebar collapse).

Watch for:
- **QML import resolution gap** (confirmed) -- pages reference `StatCard`, `DataTable`, `TimelineView`, `SecurityScoreGauge`, `ThreatChart` without any `import` statement pointing to `../../widgets` or a `qmldir` module; no `qmldir` file exists anywhere under `qml/`. This will fail at runtime unless the engine's import path is configured externally to include all subdirectories.
- **ViewModel-to-QML binding is entirely stubbed** (confirmed) -- every QML page creates a local `QtObject` with hardcoded sample data; no mechanism connects these to the C++ ViewModels. The C++ side is equally hollow (all methods return empty containers or zero).
- **No tests for the QML layer** (confirmed) -- no `qmltestrunner` tests, no visual regression tests, no ViewModel unit tests. The build passes because ViewModels compile to no-ops under `ENABLE_QT=OFF`.

## High-level view

A `ThemeManager` singleton exposes every design token (colors, spacing, radius, animations, component dimensions), and `DarkTheme.qml`/`LightTheme.qml` mirror its structure. Pages bind exclusively to `ThemeManager.*`, so theme switching propagates without per-page changes. However, `ThemeManager` uses `pragma Singleton` without a corresponding `qmldir` registration, which may prevent discovery depending on how the engine's import paths are configured.

Navigation uses a `StackView` in `main.qml` with replace transitions, driven by signals from a reworked `NavigationSidebar` that models items as grouped sections. The sidebar calculates a flat index for active-item tracking via per-delegate nested loops and `parent.parent.index` references -- fragile if the visual tree is restructured.

The ten pages follow an identical structural pattern: header, `StatCard` row, tab bar, and `Loader` switching between tab-specific `Component` blocks. Data comes from local `QtObject` stubs with hardcoded values. None will show live data without a wiring step that doesn't yet exist.

The C++ ViewModels mirror the QML stub interfaces. The `#ifdef ENABLE_QT_BUILD` split doubles each class into a QObject variant and a plain-C++ variant; the plain variant keeps CI green but introduces maintenance surface where the two halves could drift.

Widget imports are the most actionable gap. No page includes an import path to `widgets/`, no `qmldir` declares a module, and pages live two directories deep in `qml/pages/<name>/`. The pre-existing `import "../../widgets"` was removed in this change.

## Widget resolution and module structure

The 8 widgets (`SecurityScoreGauge`, `ThreatChart`, `AlertCard`, `DataTable`, `TimelineView`, `StatCard`, `ProgressRing`, `SearchBar`) live in `qml/widgets/`. Pages in `qml/pages/dashboard/DashboardPage.qml` reference `StatCard { ... }` as an unqualified type. QML's type resolution requires either: (a) the type is in the same directory, (b) an explicit `import "../../../widgets"` (relative path), or (c) a registered module with a `qmldir` file. None of these conditions are met in the diff.

The pre-existing `DashboardPage.qml` had `import "../../widgets"` which was removed in this change (the old file's line 4). Without it, the QML engine will emit `StatCard is not a type` at page load.

Similarly, `ThemeManager` is referenced as a singleton via `pragma Singleton` in `qml/themes/ThemeManager.qml`, but singletons require a `qmldir` entry (`singleton ThemeManager 1.0 ThemeManager.qml`) to be discovered. If the project uses `qrc://` resource compilation with everything flattened into one namespace, this might work implicitly, but the resource file is not part of this diff and there's no evidence of it.

## ViewModel stub pattern and integration path

Every page declares its view model as:

```qml
property var dashboardViewModel: QtObject {
    property real securityScore: 87
    ...
}
```

The C++ `DashboardViewModel` class exposes `Q_PROPERTY(double securityScore READ securityScore NOTIFY dataUpdated)` with the same field names. But nothing in `main.cpp` or anywhere in the diff performs `qmlRegisterType<DashboardViewModel>()` or uses `QML_ELEMENT`. The pages are completely self-contained demos.

The dual-class pattern means any property added to the Qt variant must be manually mirrored in the non-Qt variant -- 10 classes times two definitions is a meaningful maintenance tax.

## Sidebar flat-index calculation

The `NavigationSidebar` uses nested `Repeater`s (sections containing items). Each item delegate computes its flat index:

```qml
property int flatIdx: {
    var idx = 0
    for (var s = 0; s < parent.parent.index; s++) idx += sidebar.sections[s].items.length
    return idx + modelIndex
}
```

The `parent.parent.index` reference relies on the exact nesting depth of the visual tree. If someone wraps the delegate in an additional container, the index silently breaks. Pre-computing the flat index in the model data would eliminate this fragility.

## Missing test coverage

The change adds no tests. The ViewModels under `ENABLE_QT=OFF` compile to trivial getters returning zero/empty, so they're not meaningfully testable in that configuration. Under `ENABLE_QT=ON`, they could be tested with Qt Test, but the build infrastructure for that isn't present.

A `qmltestrunner` smoke test that loads each page would catch the widget import issue immediately.

## App.qml dual source of truth

`App.qml` grew to hold ~20 properties (alertCount, cpuUsage, scanProgress, securityScore, etc.) and utility functions. None of the pages reference these -- they all use their own local `QtObject`. This creates two sources of truth: `App.qml` says `securityScore: 85`, `DashboardPage`'s stub says `87`. When the real ViewModel bindings land, one of these layers needs to be eliminated or they'll conflict.
