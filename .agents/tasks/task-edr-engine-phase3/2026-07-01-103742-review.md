# EDR Engine: 12-component detection and response pipeline

Phase 3 introduces a full EDR stack for ThreatOne's desktop security platform. Twelve new components span process, file, registry, and memory monitoring; behavioral analysis and rule evaluation; alert management with deduplication; and specialized detectors for ransomware, lateral movement, and persistence. The orchestrator (EDREngine) wires all components together behind a clean IEDREngine interface, and MonitorEngine delegates to EDREngine for live telemetry and process enumeration. All 156 test cases pass.

**Watch for:**
- AlertManager's alert list grows without bound (confirmed) — no eviction, no cap, no persistence. Long-running agents will accumulate unbounded memory.
- `detectLateralMovement()` in EDREngine always returns an empty vector (confirmed) — no one feeds network events into LateralMovementDetection, so the public interface is dead code.
- `feedEventsToAnalysis()` is never called (confirmed) — the private method that would close the loop from file events through behavior analysis to alerts is defined but unreachable from any public path.
- Regex compilation on every rule evaluation in the hot path (confirmed) — `DetectionRulesEngine` constructs `std::regex` from scratch per event per rule, which is expensive under load.

## High-level view

The architecture delegates real monitoring to platform-specific abstractions (/proc filesystem for processes, directory polling for files, config file timestamps for "registry") and feeds events upward through a chain: monitors produce events, the BehaviorAnalyzer correlates them temporally, the DetectionRulesEngine evaluates JSON-defined rules, and the AlertManager deduplicates and publishes results to the EventBus. EDREngine orchestrates this pipeline, with MonitorEngine as a higher-level facade that adds threshold alerting on top.

The lateral movement detector is orphaned: nothing in EDREngine feeds network events into it, so `detectLateralMovement()` always returns empty. Similarly, `feedEventsToAnalysis()` — which would automatically pipe file events through the full detection chain — exists as dead code.

Resource lifecycle is the main structural concern. Monitors poll on demand (no background threads), which avoids complexity but means detection freshness depends entirely on how often callers invoke the pipeline. The alert list and incident map grow unbounded — acceptable for short-lived tests, problematic for a production agent running for days.

The test suite covers all components individually but skips the cross-component data flow (events flowing from FileMonitor through BehaviorAnalyzer to AlertManager to IncidentTimeline).

## EDREngine orchestration and the missing feedback loop

EDREngine owns all 12 sub-components by value. The design assumes callers will drive the event pipeline by calling individual detection methods. There is no internal feedback loop: `feedEventsToAnalysis()` — the method that would pipe FileMonitor events through BehaviorAnalyzer and the rules engine — is never called from any public method. The `evaluateRules()` public method does feed individual events into the analyzer, but it requires the caller to construct and pass each event explicitly. The automatic detection pipeline only partially works:

```
FileMonitor.poll() -> FileMonitor.getEvents() [stored in ring buffer]
                              |
                              v (only if caller manually calls detectRansomware)
                    RansomwareDetection.analyzeFileOperation()
                              |
                              v
                    AlertManager.generateAlert()
```

The `BehaviorAnalyzer` receives events only when `evaluateRules()` is called with explicit event data. File events collected by `FileMonitor` are never automatically forwarded to it.

## Alert deduplication and unbounded storage

AlertManager deduplicates by `source|description` key within a configurable time window (default 60s). The deduplication check iterates the full alert list on every `generateAlert()` call — O(n) scan with no index.

The `alerts_` vector only grows. There is no maximum capacity, no TTL, no pruning. For a desktop agent that might run for weeks, alert accumulation will grow proportionally to event volume. The same applies to `IncidentTimeline::incidents_` and `RegistryMonitor::events_`.

## FileMonitor ring buffer vs. RegistryMonitor unbounded list

FileMonitor uses a fixed-size ring buffer (default 1024 entries) for event storage, which naturally bounds memory. RegistryMonitor uses an unbounded `std::vector<RegistryEvent>`. If the system watches a frequently-changing config file, the registry event list grows without limit. This asymmetry means one monitor is production-safe and the other is not under sustained load.

## DetectionRulesEngine regex compilation cost

When a rule condition uses `operator: "regex"`, the engine constructs a `std::regex` on every evaluation:

```cpp
std::regex re(condition.value);
return std::regex_search(fieldValue, re);
```

`std::regex` construction is expensive (microseconds to milliseconds depending on pattern complexity). With N rules containing regex conditions evaluated against high-frequency events, this becomes a throughput bottleneck. Pre-compile patterns at rule load time and store the compiled regex alongside the `RuleCondition`.

## ProcessMonitor double-reads /proc/[pid]/cmdline

`readProcessInfo` opens `/proc/[pid]/cmdline` twice — once via `std::getline` (which reads nothing useful due to null separators), then again in binary mode to reassemble null-separated arguments. The first read is wasted I/O; only the binary read is needed. With hundreds of processes, this doubles the syscall cost for command-line reading.

## LateralMovementDetection is disconnected

EDREngine's `detectLateralMovement()` implementation is:
```cpp
return {};
```

No code in the system constructs `NetworkEvent` objects or calls `analyzeConnection()`. The component exists in isolation. `detectCredentialAccess()` and `detectRemoteExecution()` are available but never wired into the FileMonitor or ProcessMonitor event flow.

## Test coverage gaps

Not tested:
- Cross-component flow (events propagating from monitors through analyzers to alerts)
- Concurrent access from multiple threads
- `feedEventsToAnalysis()` (never called, so untestable via public API)
- MonitorEngine threshold alerting with real telemetry spikes
- `EDREngine::detectLateralMovement()` doing anything useful
- RegistryMonitor with actual file system changes during test
- Memory pressure from long-running alert/incident accumulation
