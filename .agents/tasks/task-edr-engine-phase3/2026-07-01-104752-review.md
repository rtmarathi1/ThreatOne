# EDR Engine: 12-component detection pipeline (v2)

Phase 3 delivers a full EDR stack — 12 components spanning process/file/registry/memory monitoring, behavioral correlation, rule evaluation, alert management, and specialized detectors for ransomware, lateral movement, and persistence. All v1 concerns have been structurally addressed: `feedEventsToAnalysis()` is now called from `getFileEvents()`, lateral movement detection is wired via credential-access and remote-execution checks, AlertManager enforces a 10,000-alert cap, RegistryMonitor uses a ring buffer, DetectionRulesEngine pre-compiles regex at load time, and IncidentTimeline prunes resolved incidents.

**Watch for:**
- Repeated event feeding on every `getFileEvents()` call (confirmed) — `feedEventsToAnalysis()` re-processes the entire ring buffer each invocation, feeding duplicates into BehaviorAnalyzer and re-evaluating rules against already-processed events.
- `LateralMovementIndicator.severity` never set (confirmed) — all returned indicators leave the severity field as empty string, despite the struct declaring it and EDREngine hardcoding "high" in the alert.
- `LateralMovementIndicator.timestamp` misused as description (confirmed) — the field intended for ISO timestamps stores human-readable sentences like "Credential file access: /etc/shadow by cat".

**Verdict**: NEEDS_CHANGES

## High-level view

The v1 structural gaps (dead methods, unbounded storage, uncompiled regex) are resolved. The architecture now forms a complete pipeline: monitors produce events into ring buffers, EDREngine orchestrates polling and feeds events through BehaviorAnalyzer and DetectionRulesEngine, and AlertManager deduplicates and publishes to the EventBus with a 10k cap.

The remaining issue is a design flaw in how `feedEventsToAnalysis()` interacts with the FileMonitor ring buffer. Because the buffer retains events until overwritten and `feedEventsToAnalysis()` reads the entire buffer each time, every call to `getFileEvents()` re-processes all buffered events — re-feeding them into the BehaviorAnalyzer's sliding window and re-evaluating them against all loaded rules. Under sustained file activity, this means 1,024 events processed on every call, with the same events accumulating as duplicates in the analyzer's deque until its window evicts them.

The lateral movement wiring is functional but has field-level bugs: the `severity` field is never populated on returned indicators, and `timestamp` stores descriptive text instead of a time value.

<summary>Issues (4)</summary>

1. **Duplicate event feeding per call** — `feedEventsToAnalysis()` processes every event currently in the FileMonitor ring buffer, not just new events since the last call. Track a high-water mark or cursor so each event is fed exactly once.
2. **LateralMovementIndicator.severity unset** — `detectCredentialAccess()` and `detectRemoteExecution()` never assign the `severity` field. Callers (including `EDREngine::detectLateralMovement`) receive indicators with empty severity, creating inconsistency with the "high" severity used in the companion alert.
3. **LateralMovementIndicator.timestamp misused** — The timestamp field stores human descriptions (e.g., "Credential file access: /etc/shadow by cat") rather than time values. Either rename the field to `description` or populate it with an actual timestamp.
4. **AlertManager O(n) dedup scan at capacity** — `isDuplicate()` scans up to 10,000 alerts linearly on every `generateAlert()` call. Under high event volume this becomes a throughput bottleneck. Consider indexing `dedupKey` in an `unordered_map` pointing to the most recent alert with that key.

## Duplicate event feeding through the analysis pipeline

`getFileEvents()` calls `feedEventsToAnalysis()`, which reads `fileMonitor_.getEvents()` — returning all events currently in the 1,024-slot ring buffer — and feeds each one into `behaviorAnalyzer_.addEvent()` and `rulesEngine_.evaluate()`. The ring buffer is not cleared after reading. On the next `getFileEvents()` call, the same buffer contents (minus any overwritten by new events) are re-processed.

```
Call 1: buffer has events [A, B, C] → feeds A, B, C to analyzer
Call 2: buffer has events [A, B, C, D] → feeds A, B, C, D to analyzer (A,B,C duplicated)
```

The BehaviorAnalyzer's sliding window (default 1000) will accumulate these duplicates, inflating event counts and potentially triggering false-positive pattern matches. The ransomware detector, for example, counts file renames exceeding a threshold — duplicated events would hit that threshold faster.

Fix: maintain a `lastProcessedIndex_` cursor in EDREngine and only process events beyond that index, or clear a separate "pending analysis" queue after processing.

## LateralMovementIndicator field misuse

Both `detectCredentialAccess` and `detectRemoteExecution` construct indicators as:

```cpp
indicator.timestamp = "Credential file access: " + filePath + " by " + processName;
// indicator.severity is never assigned — remains ""
```

The struct's `severity` field (added in this PR to `IEDREngine.h`) carries no value through the detection path, even though `EDREngine::detectLateralMovement` passes `"high"` to the alert manager independently. Any downstream consumer relying on the indicator's own severity gets an empty string.

## AlertManager deduplication at scale

The `isDuplicate()` method iterates the full `alerts_` vector (up to `maxCapacity_` = 10,000) comparing `dedupKey` strings and checking time windows. At 10k alerts with frequent `generateAlert()` calls (one per file event per rule match in `feedEventsToAnalysis`), this is O(n) string comparison on every alert generation. An `unordered_map<string, steady_clock::time_point>` keyed by dedupKey would make this O(1) amortized.

This is not a correctness bug — deduplication works — but it imposes a performance ceiling on the detection pipeline's throughput.
