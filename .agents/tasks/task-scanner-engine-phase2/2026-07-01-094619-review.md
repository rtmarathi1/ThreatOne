# Scanner Engine with hash-based detection, quarantine, and scheduling

The Phase 1 ScanEngine stub has been replaced with a full orchestrator that ties together ten new components: a custom SHA256 implementation (FIPS 180-4), an in-memory signature database with O(1) hash lookup, a signature matcher, a YARA engine interface (stub backend), a thread-pool scan scheduler, recursive file enumeration with glob exclusions, atomic progress tracking, a quarantine manager with metadata persistence, and a scan history layer using the existing database models. The build was also fixed across ~20 modules by correcting CMakeLists.txt source references and adding missing dependencies.

**Watch for:** Data race in `processFile` when the scheduler's thread pool pushes findings into an `ActiveScan` (confirmed). `startScan` blocks the caller even on the large-scan path, defeating the purpose of the thread pool (confirmed). The QuarantineManager does not strip file permissions on quarantined files, leaving them executable (confirmed).

## High-level view

The ScanEngine orchestrator is the coordination point: it enumerates files, decides whether to process inline or via the scheduler, matches each file's SHA256 hash against the signature database, and publishes SecurityEvent notifications through the EventBus. The design splits into detection (FileHasher + SignatureDatabase + SignatureMatcher), scheduling (ScanScheduler thread pool), lifecycle (ScanProgress atomics, start/stop/pause), and response (QuarantineManager file isolation, ScanHistory database persistence).

The SHA256 implementation follows FIPS 180-4 directly — a deliberate choice to avoid an OpenSSL dependency, trading performance and hardware acceleration for zero external crypto dependencies. BLAKE3 is explicitly a placeholder that produces deterministic but non-standard output.

The scan lifecycle has a concurrency gap: `startScan` is synchronous even when it delegates to the thread pool. For scans over 100 files it submits a batch to the scheduler, then busy-waits (10ms poll) until completion. Additionally, `processFile` pushes to `scan.findings` under `scansMutex_`, but the scheduler's worker threads call `processFile` concurrently — this is a lock granularity mismatch that risks contention and potential deadlock with `getScanStatus`.

The QuarantineManager moves files and writes JSON metadata but does not modify file permissions on the quarantined copy. On a multi-user system, a quarantined executable remains executable.

ScanHistory uses the `target` field as a surrogate key for scan lookups, which is fragile if the same target is scanned more than once.

Test coverage spans six test files covering the core detection pipeline, quarantine lifecycle, file enumeration, and the end-to-end orchestrator. No tests exercise the scheduler under real concurrency, the pause/resume flow, or the ScanHistory database layer.

## Data race and lock granularity in processFile

When called from the scheduler, multiple threads execute `processFile` concurrently on the same `ActiveScan` instance. The method locks `scansMutex_` when pushing to `scan.findings`:

```cpp
{
    std::lock_guard<std::mutex> lock(scansMutex_);
    scan.findings.push_back(finding);
}
```

This same mutex guards `activeScans_` in `getScanStatus`, `getScanResults`, `startScan`, `stopScan`, and `pauseScan`. If any of these are called while workers hold the mutex, they block. The scheduler path submits a batch and enters a poll loop — if `getScanStatus` is called from another thread during the scan, it contends on the same lock that workers need for each threat finding.

The fix: give `ActiveScan` its own mutex for the findings vector, separate from the map-level `scansMutex_`.

## startScan is synchronous despite the thread pool

The implementation blocks regardless of path:

```cpp
// Larger scan - use scheduler
scheduler_.submitBatch(files, [this, scanPtr](...) { ... });

// Wait for completion (simple polling)
while (!scan->progress->isComplete() && scan->status == ScanStatus::Running) {
    std::this_thread::sleep_for(std::chrono::milliseconds(10));
    ...
}
```

For a desktop security product, this means whatever calls `startScan` freezes for the entire scan duration. The inline path (files <= 100) processes every file in a for-loop before returning.

## Quarantine file permissions

```cpp
std::filesystem::rename(filePath, entry.quarantinedPath, ec);
```

If the original was `chmod 755`, the quarantined copy is still `chmod 755`. A script traversing the quarantine directory could execute it. Standard AV practice: strip all permissions and optionally encrypt/XOR the content to render it inert.

## Hard-coded scan targets

`getTargetsForType` returns literal paths:

```cpp
case ScanType::Quick:
    targets = {"/usr/bin", "/usr/sbin", "/home"};
    break;
case ScanType::Full:
    if (!config.targets.empty()) { ... }
    else { targets = {"/"}; }
    break;
```

This is Linux-specific. A Full scan defaulting to `/` on a production system with no explicit targets would be catastrophic for performance. The `Memory` scan type targeting `/proc` is partially mitigated by the enumerator filtering for regular files, but the intent (scanning process memory) isn't achievable through file enumeration.

## ScanHistory surrogate key fragility

```cpp
auto scanResults = Database::Models::Scan::findWhere(
    conn, "target", Database::BoundParam::text(record.target));
```

Two scans on the same directory produce ambiguous lookups. The scan ID should be the primary lookup key.

## Test coverage gaps

Not tested: the ScanScheduler under concurrent load (all test scans use <= 3 files, which takes the inline path), pause/resume behavior, the ScanHistory database layer, YARA rule loading and matching (even at the stub level), error paths when file hashing fails mid-scan, and the QuarantineManager cross-filesystem fallback (copy-then-delete) path.
