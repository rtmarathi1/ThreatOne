# Scanner Engine review fixes: per-scan mutex, YARA wiring, quarantine hardening, configurable paths (v2)

The v1 review flagged 8 issues; 4 were prioritized for immediate fix. This commit adds a per-scan `findingsMutex` to eliminate the data race on the findings vector, wires `yaraEngine_.matchFile()` into `processFile`, strips file permissions after quarantine move, and introduces `setQuickScanPaths`/`setFullScanPaths` to make scan targets configurable with platform-aware defaults. Build compiles cleanly and all tests pass.

**Watch for:** Lock ordering inversion in `getScanResults` â€” it holds `scansMutex_` while acquiring `findingsMutex`, but `processFile` acquires only `findingsMutex` (confirmed, low risk in practice but technically invertible if the design evolves). The YARA engine wiring gates on `ruleCount() > 0`, but the stub `matchFile` always returns empty matches, so this path is never exercised by any test (confirmed). Full scan still defaults to `/` (or `C:\`) when no targets and no configured defaults are provided, which is a latent footgun for production deployments (confirmed, mitigated by the new setter).

**Verdict**: APPROVED

## High-level view

The data race fix separates `findingsMutex` (per-scan, guards findings vector) from `scansMutex_` (guards the map). `getScanResults` nests locks (`scansMutex_` -> `findingsMutex`), which is safe today because `processFile` never acquires `scansMutex_`, but the ordering invariant is implicit and fragile if the design evolves.

The YARA wiring gates on `status() == Ready && ruleCount() > 0` before calling `matchFile`, but since the stub always returns empty and no test loads YARA rules before scanning, this code path is dead in CI.

Quarantine permission stripping uses `perm_options::replace` with `owner_read` (0400). However, `restore` does not restore original permissions â€” the restored file retains 0400, creating a behavioral asymmetry between quarantine and restore.

The configurable scan paths override platform defaults via setters, and `/home` was removed from Quick scan defaults. Full scan still falls through to `/` when unconfigured with no explicit targets.

## Lock ordering and the per-scan mutex

In `getScanStatus`, the implementation takes a shared_ptr copy under `scansMutex_`, releases it, then independently locks `findingsMutex` â€” no nesting. `getScanResults` takes a different path:

```cpp
std::lock_guard<std::mutex> lock(scansMutex_);
for (const auto& [id, scan] : activeScans_) {
    // ...
    {
        std::lock_guard<std::mutex> findingsLock(scan->findingsMutex);
        result.findings = scan->findings;
    }
}
```

The ordering invariant (`scansMutex_` first, `findingsMutex` second, never reversed) is implicit. If a future change adds code that locks `scansMutex_` while holding `findingsMutex` â€” perhaps to update scan status from within a findings handler â€” the result is deadlock.

## YARA wiring gap

The gate logic in `processFile`:

```cpp
if (yaraEngine_.status() == YaraEngineStatus::Ready && yaraEngine_.ruleCount() > 0) {
    auto yaraResult = yaraEngine_.matchFile(file);
    ...
}
```

No test loads and compiles YARA rules before running a scan. The engine tests load only hash-based signatures. To verify this doesn't regress, a test should call `yaraEngine().loadRulesFromString(...)`, `yaraEngine().compileRules()`, then run a scan â€” confirming the gate evaluates to true and the stub executes without error.

## Quarantine restore asymmetry

```cpp
std::filesystem::permissions(entry.quarantinedPath,
    std::filesystem::perms::owner_read,
    std::filesystem::perm_options::replace, ec);
```

Quarantine sets 0400. `restore` moves the file back without reverting permissions. A quarantined-then-restored executable becomes non-executable. For a security product this may be intentional (fail-closed), but the original permissions are not recorded in the metadata JSON, making automatic restoration impossible even if desired later.
