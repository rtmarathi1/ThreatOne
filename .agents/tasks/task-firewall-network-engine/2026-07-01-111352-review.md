# Firewall and network security engine (Phase 4)

Adds a full firewall engine (RuleEngine, RulePersistence, ConnectionTracker, PacketFilter) and eight network security components (IPBlocklist, GeoIPDatabase/GeoBlocker, DNSFilter, BandwidthMonitor, IntrusionDetector, NetworkIsolation, ConnectionLogger, ApplicationControl). Both the FirewallManager and NetworkManager stubs are replaced with functional implementations that own and wire these subsystems. ~3,950 lines across 48 files, backed by ~100 new test cases covering each component.

**Watch for:** PacketFilter misidentifies matched rules by skipping port/app checks in its secondary match loop (confirmed). IntrusionDetector alerts vector grows unbounded with no capacity limit (confirmed). `pruneOldEntries` is dead code—never called (confirmed).

## High-level view

The firewall subsystem layers RuleEngine (rule storage, CIDR matching, priority sort), PacketFilter (evaluation delegation + statistics), and FirewallManager (composition and interface exposure). RulePersistence bridges to the database ORM but is not wired into the manager—rules only live in memory until persistence is manually invoked with a connection.

The network subsystem takes a "bag of tools" approach: NetworkManager owns eight independent security components and exposes them by reference. Only `dnsLookup` checks DNSFilter and `ping` checks IPBlocklist and NetworkIsolation. The remaining components (IntrusionDetector, BandwidthMonitor, ConnectionLogger, ApplicationControl) are accessible via getters but not consulted during actual traffic operations.

IntrusionDetector is the weakest component: alerts accumulate indefinitely, `pruneOldEntries` exists but is never invoked, and detection functions add one timestamp per call rather than per actual network event, making thresholds dependent on call frequency rather than real traffic volume.

FirewallManager adds an outer mutex used only for `nextRuleId_` generation while `getRules`/`enableRule`/`disableRule` bypass it. Not a correctness bug (RuleEngine's internal lock provides protection), but unnecessarily complex—`std::atomic<int>` would eliminate the outer mutex.

<details>
<summary>Issues (7)</summary>

1. **PacketFilter matched-rule identification is incomplete** — The secondary loop in `filterPacket` that finds the matched rule ID skips port range, single-port, and application path checks. It can attribute the action to the wrong rule. Mirror the full matching logic from `RuleEngine::matchesRule` or have `evaluatePacket` return the matched rule directly.
2. **IntrusionDetector alerts vector is unbounded** — Every alert is pushed to `alerts_` with no capacity cap. Under sustained attack traffic this will exhaust memory. Add a ring buffer or max-size with eviction.
3. **`pruneOldEntries` is dead code** — The method exists but is never called from any code path. Either call it periodically (e.g., inside `evaluate`/`detectPortScan`) or remove it.
4. **SYN flood / brute force detection counts calls, not events** — `detectSYNFlood` adds one timestamp per invocation regardless of how many actual SYN packets occurred. A caller must invoke it once per packet for thresholds to be meaningful. The API suggests `count` is a threshold parameter but the implementation just always adds one entry.
5. **RulePersistence is not integrated into FirewallManager** — Rules added through the manager exist only in memory. Restarting loses all rules. Either wire persistence into add/remove or document the manual persistence requirement in the interface.
6. **BandwidthMonitor rate limiting prunes inside a `const` method** — `checkRateLimit` is `const` but calls `getCurrentWindowBytes` which mutates the `mutable` windowEntries vector. A read-only query has side effects (pruning old entries), which could surprise callers expecting idempotent reads.
7. **NetworkManager sub-components are not integrated into traffic flow** — IntrusionDetector, BandwidthMonitor, ApplicationControl, and ConnectionLogger are accessible but none are consulted during actual send/receive operations. Without hook points these provide no automatic protection.

## PacketFilter matched-rule divergence from RuleEngine

`PacketFilter::filterPacket` calls `ruleEngine_.evaluatePacket(packet)` to get the action, then runs a second loop over all rules to determine which rule produced that action. The second loop checks source IP, dest IP, protocol, and direction—but omits port matching (both single-port and range) and application path matching. If two rules differ only by port, the filter will report the first rule in priority order as the match even when the actual match came from a different rule.

```cpp
// PacketFilter.cpp — secondary match loop (incomplete)
bool sourceMatch = rule.sourceIP.empty() ||
    RuleEngine::matchCIDR(packet.sourceIP, rule.sourceIP, ...);
bool destMatch = rule.destIP.empty() ||
    RuleEngine::matchCIDR(packet.destIP, rule.destIP, ...);
bool protoMatch = rule.protocol == Protocol::Any || rule.protocol == packet.protocol;
bool dirMatch = rule.direction == packet.direction;
// Missing: port, portRange, applicationPath
```

The cleanest fix is to have `RuleEngine::evaluatePacket` return both the action and the matched rule ID, eliminating the duplicate logic.

## IntrusionDetector resource management

The `alerts_` vector grows without bound. Four code paths append alerts (`evaluate`, `detectPortScan`, `detectSYNFlood`, `detectBruteForce`) and nothing trims them except an explicit `clearAlerts()` call from the consumer.

`pruneOldEntries()` is defined and correctly cleans `portScanTracker_`, but it is private and never called. The `synFloodTracker_` and `bruteForceTracker_` maps are pruned per-call within their respective detect methods, but `portScanTracker_` relies solely on this dead method for cleanup beyond the 60-second window in `detectPortScan`. If `detectPortScan` is called for IP "A" and then never called again for "A", those entries remain forever.

## Detection threshold semantics

`detectSYNFlood(destIP, count, windowSeconds)` pushes a single timestamp on every call and fires when `timestamps.size() >= count`. The threshold is "number of times the method was called within the window" not "number of SYN packets observed." If the caller batches 1000 packets and calls the method once, it registers as 1 event. The semantics require the caller to invoke exactly once per network event, which is undocumented and non-obvious from the API signature where `count` reads like "how many events constitute a flood."

## Test coverage gaps

Not tested: RulePersistence (requires a database connection mock), FirewallManager as an integration (tests hit RuleEngine and PacketFilter directly), NetworkManager integration (ping/DNS with blocklist checks), ConnectionLogger ring-buffer wrap-around behavior, and concurrent access to any component.
