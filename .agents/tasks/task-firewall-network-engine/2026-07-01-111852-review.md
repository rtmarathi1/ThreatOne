# Firewall and network security engine (v2)

The v1 review raised 7 issues. Four confirmed bugs have been fixed: RuleEngine now returns `EvalResult{action, matchedRuleId}` eliminating PacketFilter's broken secondary match loop, IntrusionDetector caps alerts at 10,000 with oldest-first eviction, `pruneOldEntries` is called from `evaluate()` and `detectPortScan()`, and `BandwidthMonitor::checkRateLimit` is no longer `const`. Three design-intent items (detection threshold semantics, RulePersistence not wired in, sub-components not auto-consulted) were addressed with documentation clarifying the intended caller-driven usage model.

**Watch for:** IntrusionDetector's `synFloodTracker_` and `bruteForceTracker_` maps grow unbounded across unique keys (confirmed). FirewallManager double-locking pattern acquires its own mutex then calls into RuleEngine which acquires a second mutex, creating unnecessary contention on `addRule`/`removeRule`/`blockIP`/`allowIP` (confirmed). Alert eviction via `erase(begin())` on a vector is O(n) per insertion at capacity (confirmed).

## High-level view

IntrusionDetector's alert cap and pruning fix address the most critical v1 concerns, but the fix introduces an O(n) eviction cost because `alerts_` is a `std::vector` and eviction erases the front element. At 10,000 entries this shifts ~80KB per alert insertion once saturated. A `std::deque` or ring buffer would eliminate this.

The tracker maps (`synFloodTracker_`, `bruteForceTracker_`) prune timestamps within each key but never remove empty keys from the map itself. Under a distributed attack hitting thousands of unique IPs, these maps grow without bound. `pruneOldEntries` only cleans `portScanTracker_`; the other two maps have no cross-key cleanup.

FirewallManager wraps `addRule`, `removeRule`, `blockIP`, and `allowIP` in its own mutex before calling RuleEngine methods that independently acquire RuleEngine's mutex. This double-lock is safe (consistent order) but serializes unrelated operations unnecessarily. Converting `nextRuleId_` to `std::atomic<int>` would eliminate the outer mutex entirely.

3. **FirewallManager unnecessary double-lock** — The manager's mutex protects only `nextRuleId_` generation, but wraps entire `addRule`/`removeRule`/`blockIP`/`allowIP` calls including the RuleEngine call that takes its own lock. Use `std::atomic<int>` for the ID counter and drop the outer mutex.
4. **Alert eviction not tested** — No test exercises the kMaxAlerts=10000 boundary or verifies that oldest alerts are evicted first. Add a test with a smaller cap or a direct test that pushes past 10,000.

## Tracker map key accumulation

`detectSYNFlood` indexes `synFloodTracker_` by `destIP` and prunes old timestamps within that key's vector, but never checks whether the vector is now empty and should be removed from the map. The same pattern applies to `bruteForceTracker_` (keyed by `destIP:port`). In contrast, `pruneOldEntries` correctly erases empty entries from `portScanTracker_`:

```cpp
// pruneOldEntries — correctly removes empty keys from portScanTracker_
if (it->second.empty()) {
    it = portScanTracker_.erase(it);
}
```

Neither `detectSYNFlood` nor `detectBruteForce` has equivalent cleanup. Under a DDoS scenario with rotating source/dest IPs, these maps grow indefinitely. The fix is to add the same empty-key erasure after pruning timestamps in both methods, or to extend `pruneOldEntries` to cover all three maps and call it from all detection paths.

## Vector eviction performance

`appendAlert` evicts from the front of a `std::vector<IDSAlert>`:

```cpp
void IntrusionDetector::appendAlert(const IDSAlert& alert) {
    if (alerts_.size() >= kMaxAlerts) {
        alerts_.erase(alerts_.begin()); // O(n) shift
    }
    alerts_.push_back(alert);
}
```

Each `IDSAlert` contains multiple `std::string` fields (~7 strings per alert). At capacity, every new alert triggers a memmove of 10,000 alert objects. Under sustained attack traffic where alerts fire continuously, this becomes a hot path. Switching to `std::deque<IDSAlert>` gives O(1) `pop_front` with no other API changes since `getAlerts()` already copies to a vector.

## FirewallManager lock contention

`FirewallManager::addRule` acquires `mutex_`, increments `nextRuleId_`, then calls `ruleEngine_.addRule()` which internally acquires `RuleEngine::mutex_`. The outer lock is held for the duration of the inner lock:

```cpp
bool FirewallManager::addRule(const FirewallRule& rule) {
    std::lock_guard<std::mutex> lock(mutex_);   // outer
    // ...
    bool result = ruleEngine_.addRule(ruleToAdd); // acquires RuleEngine::mutex_
    return result;
}
```

`getRules()`, `enableRule()`, and `disableRule()` bypass the outer mutex entirely, going directly to `ruleEngine_`. The asymmetry means the outer mutex doesn't provide any consistency guarantee between `nextRuleId_` and the rule set — another thread can observe the new rule via `getRules()` before the outer lock is released. The mutex protects nothing beyond the integer increment. An `std::atomic<int> nextRuleId_` would be simpler, lock-free, and eliminate the nested-lock pattern.

## Test coverage for eviction

No test hits the `kMaxAlerts` boundary or verifies oldest-first eviction semantics. The constant is 10,000 which makes a brute-force test slow, but a focused test could push alerts via a content-matching rule in a tight loop, or the constant could be made configurable for testing. Without this coverage, regressions in eviction logic would go undetected.
