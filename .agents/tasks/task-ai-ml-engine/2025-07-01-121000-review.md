# AI/ML Engine: from-scratch threat analysis pipeline (v2)

This change replaces stub implementations of AIEngine and AnalyticsEngine with a full 12-component ML inference and behavioral analytics pipeline, all implemented from scratch in C++20. The architecture wires FeatureExtractor, ThreatClassifier, AnomalyDetector, RiskScoringEngine, NLPProcessor, RecommendationEngine, AutoInvestigation, and PredictionEngine through an AIEngine orchestrator. The v2 revision fixes the Welford variance formula in BehaviorAnalyticsEngine, pre-compiles regex patterns in NLPProcessor, removes the global mutex from AIEngine (relying on per-subsystem synchronization), and caps NLP input size to 1MB.

**Watch for:** `matchPattern` (public API) still runs `std::regex` against unbounded caller-provided text — the 1MB cap only protects the internal `summarizeLog` path (confirmed). The `analyzePattern` method's add-then-detect sequence on AnomalyDetector is not atomic, so concurrent callers can interleave their data points with each other's detection calls (confirmed). The isolation forest `c(n)` formula uses `evaluatedFeatures` (number of features with baseline data) rather than sample count, producing unstable thresholds as feature coverage varies (confirmed).

**Verdict**: APPROVED

## High-level view

The global mutex removal is sound — FeatureExtractor is genuinely stateless, ThreatClassifier and AnomalyDetector carry their own locks, and the remaining subsystems (RecommendationEngine, AutoInvestigation, PredictionEngine) are const-only with no mutable state. The trade-off is that `analyzePattern`'s multi-step interaction with AnomalyDetector (add points then detect) is no longer atomic, creating a data-integrity race under concurrent load.

The Welford fix in BehaviorAnalyticsEngine is correct: `buildBaseline` stores `M2 = populationVariance * n` (i.e., `sum((x_i - mean)^2)`) and `updateBaseline` applies the standard delta-delta2 recurrence. The transition between batch initialization and incremental updates is seamless because both operate on the same M2 quantity.

NLPProcessor's internal `summarizeLog` path is protected (pre-compiled patterns, 1MB cap, truncation reporting), but the public `matchPattern` API accepts arbitrary-size text with no guard — same stack overflow risk as before, just on a different code path.

<summary>Issues (3)</summary>

1. **matchPattern unbounded input** — The public `matchPattern(text, patterns)` API applies `std::regex` to text of arbitrary size. Unlike `summarizeLog`, there is no size cap. Large inputs risk stack overflow from libstdc++ regex. Add a size check or document the caller's responsibility.
2. **analyzePattern non-atomic add-detect sequence** — `analyzePattern` calls `addDataPoint` for each feature then `detectIsolationForest`. Under concurrent access, another thread's data points could be inserted between the add and detect phases, polluting the detection baseline. This is a data-integrity race, not a memory-safety race.
3. **Isolation forest c(n) uses feature count instead of sample count** — `computeIsolationScore` computes `c(n)` using `evaluatedFeatures` (typically 5-10) rather than the actual sample count stored in each feature's statistics. This makes the anomaly threshold unstable as different inputs exercise different numbers of features.

## matchPattern still vulnerable to large inputs

The `summarizeLog` 1MB cap (fix for v1 issue #5) protects the internal code path, but the public `matchPattern` API has no equivalent guard:

```cpp
std::vector<std::string> NLPProcessor::matchPattern(
    const std::string& text,
    const std::vector<std::string>& patterns) const {
    for (const auto& pattern : patterns) {
        std::regex re(pattern);
        auto begin = std::sregex_iterator(text.begin(), text.end(), re);
        // ...
    }
}
```

Any caller passing a multi-megabyte string hits the same `std::regex` stack overflow risk that `summarizeLog` now avoids. Either document that callers must pre-truncate, or add a `kMaxMatchInputBytes` guard analogous to `kMaxSummarizeInputBytes`.

## Non-atomic add-detect in analyzePattern

With the global mutex removed, `AIEngine::analyzePattern` executes:

```cpp
for (const auto& [name, value] : featureVector) {
    anomalyDetector_.addDataPoint(name, value);  // acquires/releases mutex per call
}
auto anomalyResult = anomalyDetector_.detectIsolationForest(featureVector);  // acquires mutex
```

Each `addDataPoint` and the final `detectIsolationForest` acquire AnomalyDetector's internal mutex independently. If two threads call `analyzePattern` concurrently, thread A's data points become part of the baseline before thread B's detection runs, and vice versa. Each detection potentially includes the other call's newly-added points, producing cross-contaminated baselines. The effect is that anomaly scores become less reliable under concurrent load — detection that should flag a single anomalous sample may instead see it blended into a shifted baseline.

## Isolation forest normalization semantics

The `c(n)` normalization in `computeIsolationScore`:

```cpp
double cn = 2.0 * (std::log(static_cast<double>(evaluatedFeatures)) + 0.5772) -
            2.0 * (static_cast<double>(evaluatedFeatures) - 1.0) /
            static_cast<double>(evaluatedFeatures);
```

In the standard isolation forest paper, `n` in `c(n)` is the number of training samples (i.e., `stats.count`), not the number of features evaluated. Using feature count (typically 5-10) produces very small `cn` values, which amplifies the anomaly score toward 1.0. The threshold of 0.6 was likely tuned against this behavior, so it functions for a fixed feature set, but the score's meaning shifts if a different set of features has baseline data — a sample evaluated against 3 features gets a different normalization than one evaluated against 8.

## ThreatClassifier spurious mutex

`ThreatClassifier::classifyFromVector` acquires a `mutable std::mutex` even though none of the scoring methods modify any state — the class has no mutable data members beyond the mutex itself. This serializes all classification calls unnecessarily. Not a correctness issue, but a latency cost under concurrent classification workloads that could be eliminated by removing the lock.
